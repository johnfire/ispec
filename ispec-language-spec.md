# The .ispec Language
## A Behavioral Specification System for AI-Assisted Software Development

*Version 0.1 — Draft Specification*
*Origin: March 2026*

---

## Foreword — How This Started

This language did not begin with a syntax diagram or a compiler design. It began with a question asked in a late-night brainstorming session:

*What would a programming language look like if it were designed for AI to write, rather than humans?*

The conversation went deeper than expected. The realization emerged quickly that the question was not really about syntax. It was about the fundamental relationship between human intent and running software — a relationship that has been broken since the beginning of computing, and that AI makes it possible to fix properly for the first time.

The problem is old and well understood: humans describe what they want in natural language. Developers translate that into code. The translation loses information, introduces assumptions, and is invisible to the people who made the original request. Documentation is written to bridge the gap, then immediately begins drifting from reality because humans are human — there is never enough time, things change faster than anyone can track, and the documentation update always gets deferred until it never happens.

The result: software that does something slightly different from what was asked, documentation that describes something that no longer exists, and clients who cannot verify whether what was built matches what they approved.

The .ispec system is an attempt to solve this structurally rather than procedurally. Not with better discipline or better tooling that still requires human maintenance, but by removing the human from the synchronization loop entirely.

The core insight: if the specification and the documentation are generated from the same source by an AI, they cannot drift apart. If the client approves the specification before the compiler runs, they are approving the actual thing the system will execute from — not a proxy, not a summary, not a promise. The specification is the contract.

The long-range vision is larger: a new computing paradigm in which AI writes code in a purpose-built intermediate language, humans approve intent at the specification level, and all existing programming languages become irrelevant compilation targets on the way to machine code or WebAssembly. The .ispec system is the first step toward that vision — practical, buildable now, and designed so that every phase of its evolution is architecturally coherent with where it is eventually going.

## Part One — Architecture Overview

### The Three Layers

The system has three distinct layers, each constraining the one below it.

**Layer 0 — Intent (.intent)**

The constitutional layer. Defines why the system exists, what it values, who it serves, and what it will never do regardless of any other consideration. The AI reads this before generating anything. No specification, no compiler output, and no self-modification of the system may violate the intent layer.

**Layer 1 — Specification (.ispec)**

The behavioral layer. Defines what the system does — its data shapes, operations, business rules, failure modes, events, and queries. This is the layer humans approve. The client reads the human view. The developer reviews the formal view. Both are generated from the same source and are always identical in meaning.

**Layer 2 — Implementation (generated code)**

The execution layer. Generated by a compiler from the specification. In the current phase, this is Python, JavaScript, or similar high-level languages. In the long-range vision, this will be a dense AI-native intermediate representation that compiles directly to machine code or WebAssembly, making existing languages unnecessary.

### The Pipeline

```
Human describes intent in natural language
              ↓
AI reads .intent file, produces architecture document
              ↓
Human approves architecture document
              ↓
AI generates .ispec file (h doc + s doc)
              ↓
Human approves .ispec
  — client reads h doc (plain English)
  — developer reviews s doc (formal spec)
  — this approval is the contract
              ↓
Compiler reads s doc, generates code
              ↓
Running system
  — h doc is now living documentation
  — forever in sync with what was built
```

### The Dual Representation

Every .ispec file contains two views of the same specification:

- **h doc** — the human document. Plain English. Written for non-technical readers. Clients approve this. It is generated by the AI and maintained in sync with the s doc automatically.

- **s doc** — the specification document. Formal syntax. Written for the compiler and for technical review. The AI generates this from the same understanding that produced the h doc.

These two views cannot drift apart because they are generated simultaneously from the same source and kept in sync by the AI. This is a structural guarantee, not a process discipline.

## Part Two — File Types

### .intent — The Constitution

One per system. Defines purpose, values, stakeholders, and permanent boundaries.

```
purpose:
  Why this system exists. What it is for.
  What success looks like for the humans depending on it.

serves:
  1. primary stakeholder
  2. secondary stakeholder
  3. efficiency — always last, never first

values:
  — what this system treats as non-negotiable
  — what it will never optimize away

boundaries:
  the system will never:
    — list of permanent prohibitions

intent:
  plain English guidance for ambiguous situations
  not covered by the spec

fitness:
  a modification is an improvement if and only if:
    — all spec contracts still hold
    — all values are still satisfied
    — the system serves its purpose more effectively
    — no boundary has been crossed
```

### global.policy — System-Wide Rules

One per system. Defines rules, error hierarchy, and defaults that apply across all domains.

```
[global.policy]

  --- error hierarchy ---

  errors:
    SystemError
      DatabaseError       transient, retriable
      NetworkError        transient, retriable
      TimeoutError        transient, retriable
      ServiceUnavailable  transient, retriable

    ClientError
      ValidationError
      AuthorizationError
      NotFoundError
      ConflictError

    BusinessError
      PolicyViolation
      LimitExceeded
      StateError

  --- global policies ---

  RetryPolicy:
    maximum attempts: 3
    backoff: exponential
    initial delay: 1 second

  PaginationPolicy:
    default page size: 20
    maximum page size: 100
```

### domain.ispec — Domain Specification

One per domain. Contains h doc, domain policies, and s doc.

```
[human]
  Plain English description of this domain.
  Entities, operations, queries — all described
  in language a non-technical reader can verify.
[/human]

[policy]
  Domain-specific rules that apply within this domain.
  Override global policies where they conflict.
[/policy]

[spec]
  Formal specification. Compiler reads this.
  Entities, enums, types, operations, queries,
  events, subscriptions, and extensions.
[/spec]
```

## Part Three — The Specification Language

### Primitive Domain Types

The spec language has no primitive types. All types are domain types. The compiler maps these to appropriate language primitives.

```
DOMAIN TYPE       MEANING                  PYTHON TYPE
————————————————————————————————————————————————————————
Text              string of characters     str
Email             validated email address  str (validated)
UserId            unique user identifier   UUID
Money             monetary value           Decimal
Timestamp         point in time            datetime
Boolean           true or false            bool
Count             non-negative integer     int
PhoneNumber       validated phone number   str (validated)
Url               validated URL            str (validated)
IpAddress         network address          str (validated)
```

Custom domain types are defined with the `type` construct.

### Entity

Defines a data structure — the shape of a thing the system stores and reasons about.

**Syntax:**
```
entity EntityName
  field_name: DomainType  [constraints]
  field_name: → OtherEntity  [constraints]
  field_name: →[OtherEntity] [constraints]
  status_field: ValueA | ValueB | ValueC
  history: immutable | transient
```

**Constraints:**
```
required        must be present
optional        may be absent
unique          no two records share this value
generated       system sets this, not the user
immutable       cannot change after creation
write-only      can be set, never read (passwords)
min: N          minimum value or length
max: N          maximum value or length
pattern: "..."  must match this regular expression
default: value  value if not provided
```

**References:**
```
→ Entity        reference to a single entity (foreign key)
→[Entity]       reference to a collection of entities
```

**History:**
```
history: immutable   append-only storage — default
                     current state = latest record
                     all history permanently preserved

history: transient   standard read/write storage
                     values overwritten
                     no history preserved
                     use for: sessions, counters,
                     rate limits, cache entries
```

Default is `immutable`. The annotation is only required when opting out.

**Example:**
```
entity User
  id:           UserId         [unique, generated, immutable]
  email:        Email          [unique, required]
  password:     HashedPassword [write-only]
  display_name: Text           [required, max: 100]
  bio:          Text           [optional, max: 500]
  status:       UserStatus     [default: Active]
  created_at:   Timestamp      [generated, immutable]
  updated_at:   Timestamp      [generated]
```

### Enum

Defines a named set of values.

**Syntax:**
```
enum EnumName
  ValueA
  ValueB
  ValueC
```

**Example:**
```
enum UserStatus
  Active
  Suspended
  Deleted
```

### Type

Defines a custom domain type derived from an existing type.

**Syntax:**
```
type NewTypeName is BaseType [constraints]
```

**Example:**
```
type ProductSku  is Text    [pattern: "[A-Z]{2}-[0-9]{6}"]
type Percentage  is Count   [min: 0, max: 100]
type OrderId     is UniqueId
```

---

### Use

Imports entities or types from another domain file.

**Syntax:**
```
use EntityName from domain-file-name
use EntityA, EntityB from domain-file-name
```

**Rules:**
- Every cross-domain reference must be named explicitly
- No wildcard imports
- Use the domain file name without extension

**Example:**
```
use User from user-account
use Product, ProductSku from products
use User, Session from user-account
```

### Operation

Defines a behavioral action the system can perform. Operations do things. They are distinct from queries, which ask things.

**Syntax:**
```
operation OperationName
  inputs:
    param_name: Type

  requires:
    condition
    condition

  ensures:
    condition
    EntityName created with:
      field == value

  side_effects:
    emit EventName with:
      field == value

  returns:
    Type

  idempotent: true | false
    key: field

  failures:
    ErrorName: ErrorParent
      condition: condition description
      context:
        field: Type
      retriable: true
      retry_after: duration
```

**Sections — all optional except ensures:**

`inputs` — what the operation receives from the caller

`requires` — preconditions that must be true before the operation runs. If any precondition fails, the operation does not execute and returns an appropriate failure.

`ensures` — postconditions that will be true after the operation completes successfully. This is the core behavioral contract.

`side_effects` — things that happen outside the operation's primary concern, such as emitting events or sending notifications.

`returns` — what the operation gives back to the caller.

`idempotent` — declares that running this operation multiple times with the same inputs produces the same result. Required for `at_least_once` event handlers. The `key` field specifies what makes two calls identical.

`failures` — what can go wrong, under what conditions, and how the caller should respond. Each failure names its parent in the error hierarchy, its condition, optional context data, and whether it is retriable.

**Special values in ensures:**
```
was(field)   value of field before the operation ran
now          current timestamp, generated by system
```

**Example:**
```
operation TransferFunds
  inputs:
    source:      Account
    destination: Account
    amount:      Money
    actor:       UserId

  requires:
    source.owner == actor
    source.status == Active
    destination.status != Closed
    source.balance >= amount
    amount > 0

  ensures:
    source.balance      == was(source.balance) - amount
    destination.balance == was(destination.balance) + amount
    atomic: [source.balance, destination.balance]

  side_effects:
    emit FundsTransferred with:
      source      == source
      destination == destination
      amount      == amount
      transferred_at == now

  failures:
    NotAuthorized: AuthorizationError
      condition: actor != source.owner

    InsufficientFunds: BusinessError
      condition: source.balance < amount
      context:
        available: Money
        requested: Money

    NetworkError: NetworkError
      condition: transient
      retriable: true
```

### Query

Defines a data retrieval operation. Queries ask things. They do not change state.

**Syntax:**
```
@query QueryName
  find:    EntityName
  through: JoinEntity
  where:   condition
  and:     condition
  or:      condition
  order:   Entity.field ascending | descending
  limit:   Count
  offset:  Count
  returns: one
```

**Keywords:**

`find` — what entity to return. Required.

`through` — join entity to traverse for many-to-many relationships. Optional.

`where` — primary filter condition. Required.

`and` / `or` — additional filter conditions.

`order` — sort field and direction.

`limit` — maximum results to return.

`offset` — pagination start position.

`returns: one` — this query returns exactly one result or fails with `NotFoundError`.

Without `returns: one`, the default return is a collection, which may be empty.

**Example:**
```
@query GetAccountTransactions
  find:  Transaction
  where: Transaction.account == account
  and:   Transaction.status == Completed
  order: Transaction.created_at descending
  limit: 50

@query GetUserByEmail
  find:    User
  where:   User.email == email
  returns: one
```

### Event

Defines something that happened — a fact about the past that other domains may need to know.

Events are owned by the domain that emits them. They are immutable by definition. They carry data describing what happened.

**Syntax:**
```
event EventName
  field: Type     [constraints]
  field: → Entity [constraints]
```

**Example:**
```
event FundsTransferred
  source:         → Account  [required]
  destination:    → Account  [required]
  amount:         Money      [required]
  transferred_at: Timestamp  [required]
```

---

### Subscription

Declares that a domain responds to events from another domain.

**Syntax:**
```
use EventName from source-domain

on EventName
  trigger:  OperationName
  delivery: at_least_once | exactly_once | at_most_once
```

**Delivery semantics:**
```
at_least_once   may arrive more than once
                handler must be idempotent
                use for: emails, notifications

exactly_once    guaranteed single delivery
                use for: payments, inventory

at_most_once    may be lost, never duplicated
                use for: analytics, logging
```

**Example:**
```
use FundsTransferred from accounts

on FundsTransferred
  trigger:  SendTransferConfirmation
  delivery: at_least_once
```

### Aggregates

Reduce a collection of values to a single value. Used in requires, ensures, failures, and @extend computed.

**Syntax:**
```
count(Entity where conditions)
sum(Entity.field where conditions)
avg(Entity.field where conditions)
min(Entity.field where conditions)
max(Entity.field where conditions)
exists(Entity where conditions)
all(Entity where conditions such_that condition)
none(Entity where conditions such_that condition)
```

**Arithmetic inside aggregates:**
```
sum(Entity.fieldA * Entity.fieldB where conditions)
```

Supported operators: `*`, `+`, `-`, `/`

For complex arithmetic, define a computed field first and aggregate that.

**Type rules:**
```
sum()     field must be Money, Count, or numeric
avg()     field must be numeric, returns Decimal
min/max() field must be comparable
count()   any entity, returns Count
exists()  any entity, returns Boolean
all()     any entity with such_that, returns Boolean
none()    any entity with such_that, returns Boolean
```

**Example:**
```
count(Enrollment
      where Enrollment.course == course
      and Enrollment.status == Active)

sum(OrderItem.unit_price * OrderItem.quantity
    where OrderItem.order == order)

exists(Session
       where Session.user == user
       and Session.status == Valid)

none(Payment
     where Payment.order == order
     such_that Payment.status == Failed)
```

### Extension Constructs

Extensions handle concepts the core language does not address natively.

**@extend index — performance hints**

```
@extend index
  query: Entity by field           [frequent]
  query: Entity by field           [unique]
  query: Entity by field           [range]
  query: Entity by field, field    [frequent]
```

Tags:
```
[frequent]   high read volume, optimize aggressively
[unique]     always returns one result
[range]      queries span a range of values
[occasional] low volume, basic index sufficient
```

**@extend computed — derived fields**

```
@extend computed
  Entity.field_name = aggregate(...)
  Entity.field_name = (arithmetic expression)
```

**@extend audit — explicit audit tracking**

```
@extend audit
  track: Entity.field
  track: Entity.field
```

## Part Four — Error Hierarchy

Errors form an inheritance tree rooted in three families. The hierarchy is defined in global.policy and extended in each domain's operation failure declarations.

```
SystemError                    infrastructure problems
  DatabaseError                  transient, retriable
  NetworkError                   transient, retriable
  TimeoutError                   transient, retriable
  ServiceUnavailable             transient, retriable

ClientError                    caller did something wrong
  ValidationError                bad input
  AuthorizationError             not permitted
  NotFoundError                  does not exist
  ConflictError                  state prevents action

BusinessError                  business rule violated
  PolicyViolation                explicit rule
  LimitExceeded                  cap or quota
  StateError                     wrong lifecycle state
```

**Recoverability is inherited:**
```
SystemError    always transient, always retriable
ClientError    never retriable — fix the input first
BusinessError  never retriable — state must change first
```

Individual errors may override:
```
failures:
  TemporaryLock: BusinessError
    condition: resource.locked == true
    retriable: true
    retry_after: 30s
```

## Part Five — A Complete Example

The following is a complete .ispec file for a simple task management system. It demonstrates every major construct in a realistic context.

### The .intent File

```
purpose:
  This system helps small teams track and complete work.
  Every decision the system makes must serve the people
  doing the work, not the process of tracking it.

serves:
  1. team members    — people doing the work
  2. team leads      — people coordinating the work
  3. administrators  — people managing the system
  4. efficiency      — last, never first

values:
  — work is never lost
    no task or comment is ever permanently deleted.
    archive, do not destroy.

  — people are treated with respect
    the system does not surveil. it tracks work,
    not workers.

  — the system is honest about its state
    errors are reported clearly. the system does
    not hide failures or pretend success.

boundaries:
  the system will never:
    — permanently delete tasks or comments
    — expose one user's private notes to another user
    — send communications without explicit trigger

intent:
  when a situation is ambiguous, ask: does this
  help the person get their work done? if not,
  the answer is probably no.

fitness:
  a modification is an improvement if and only if:
    — all spec contracts still hold
    — no task or comment has been lost
    — team members can accomplish their work
      more effectively
    — no boundary has been crossed
```

### global.policy

```
[global.policy]

  --- error hierarchy ---

  errors:
    SystemError
      DatabaseError       transient, retriable
      NetworkError        transient, retriable
      TimeoutError        transient, retriable
      ServiceUnavailable  transient, retriable

    ClientError
      ValidationError
      AuthorizationError
      NotFoundError
      ConflictError

    BusinessError
      PolicyViolation
      LimitExceeded
      StateError

  --- policies ---

  RetryPolicy:
    maximum attempts: 3
    backoff: exponential
    initial delay: 1 second

  PaginationPolicy:
    default page size: 20
    maximum page size: 100

  TaskPolicy:
    maximum tasks per project: 1000
    maximum comment length: 5000 characters
    maximum task title length: 200 characters
```

### tasks.ispec

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
FILE: tasks.ispec
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[human]

  Domain: Tasks

  This domain manages the tasks that teams create,
  assign, and complete. It covers the full lifecycle
  of a task from creation to completion or archival.


  --- Entities ---

  A Project is a container for related tasks. It has
  a name, an optional description, a status, and the
  member who created it. Projects are never deleted —
  they are archived when no longer active. All changes
  to a project are permanently preserved.

  A Task belongs to one Project. It has a title, an
  optional description, a priority, a status, and
  optionally an assignee and a due date. Tasks are
  never deleted — they are archived when no longer
  needed. All changes to a task are permanently
  preserved.

  A Comment belongs to one Task. It records what a
  team member said and when. Comments are never
  deleted. All comments are permanently preserved.

  A Project can be:
    Active    — currently in use
    Archived  — no longer active, preserved for reference

  A Task can be:
    Open        — not yet started
    In Progress — being worked on
    Done        — completed
    Archived    — no longer relevant

  A Task Priority can be:
    Low
    Normal
    High
    Critical


  --- Operations ---

  Create Task:
    A team member creates a task in a project.
    The project must be active. The task starts
    as Open with Normal priority unless specified.
    A notification is sent to the assignee if one
    is provided.

  Assign Task:
    A task is assigned to a team member. The task
    must be Open or In Progress. The previous
    assignee is notified that the task has moved.
    The new assignee is notified of the assignment.

  Start Task:
    A team member begins work on a task assigned
    to them. The task must be Open.

  Complete Task:
    A team member marks their task as done.
    The task must be In Progress and assigned
    to the person completing it.

  Add Comment:
    Any team member adds a comment to any task.
    Comments cannot be empty.


  --- Queries ---

  Get Project Tasks:
    Returns all non-archived tasks in a project,
    ordered by priority (highest first) then
    by creation date (newest first).

  Get My Tasks:
    Returns all Open and In Progress tasks
    assigned to a specific team member,
    ordered by priority then due date.

  Get Task Comments:
    Returns all comments on a task,
    ordered oldest first.

  Get Overdue Tasks:
    Returns all Open and In Progress tasks
    where the due date has passed,
    ordered by due date ascending.

[/human]


[policy]

  ProjectPolicy:
    maximum active projects: 50

  TaskPolicy:
    maximum tasks per project: 1000
    maximum comment length: 5000 characters

[/policy]


[spec]

  domain Tasks

  use User from user-account


  --- enums ---

  enum ProjectStatus
    Active
    Archived

  enum TaskStatus
    Open
    InProgress
    Done
    Archived

  enum Priority
    Low
    Normal
    High
    Critical


  --- entities ---

  entity Project
    id:          ProjectId     [unique, generated, immutable]
    name:        Text          [required, max: 200]
    description: Text          [optional, max: 2000]
    status:      ProjectStatus [default: Active]
    created_by:  → User        [required, immutable]
    created_at:  Timestamp     [generated, immutable]
    updated_at:  Timestamp     [generated]

  entity Task
    id:          TaskId    [unique, generated, immutable]
    project:     → Project [required, immutable]
    title:       Text      [required, max: 200]
    description: Text      [optional, max: 5000]
    status:      TaskStatus [default: Open]
    priority:    Priority   [default: Normal]
    assignee:    → User     [optional]
    due_date:    Timestamp  [optional]
    created_by:  → User     [required, immutable]
    created_at:  Timestamp  [generated, immutable]
    updated_at:  Timestamp  [generated]

  entity Comment
    id:         CommentId [unique, generated, immutable]
    task:       → Task    [required, immutable]
    author:     → User    [required, immutable]
    body:       Text      [required, max: 5000]
    created_at: Timestamp [generated, immutable]


  --- operations ---

  operation CreateTask
    inputs:
      project:     Project
      title:       Text
      description: Text        [optional]
      priority:    Priority    [optional, default: Normal]
      assignee:    → User      [optional]
      due_date:    Timestamp   [optional]
      creator:     User

    requires:
      project.status == Active
      title is not empty
      count(Task
            where Task.project == project
            and Task.status != Archived)
        < TaskPolicy.maximum_tasks_per_project

    ensures:
      Task created with:
        project     == project
        title       == title
        description == description
        priority    == priority
        assignee    == assignee
        due_date    == due_date
        status      == Open
        created_by  == creator

    side_effects:
      emit TaskCreated with:
        task       == created Task
        project    == project
        assignee   == assignee
        created_by == creator
        created_at == now

    returns:
      Task

    failures:
      ProjectArchived: StateError
        condition: project.status != Active

      TitleEmpty: ValidationError
        condition: title is empty

      ProjectFull: LimitExceeded
        condition: active task count >=
                   TaskPolicy.maximum_tasks_per_project
        context:
          current_count: Count
          maximum:       Count


  operation AssignTask
    inputs:
      task:     Task
      assignee: User
      actor:    User

    requires:
      task.status == Open or task.status == InProgress
      task.project.status == Active

    ensures:
      task.assignee   == assignee
      task.updated_at == now

    side_effects:
      emit TaskAssigned with:
        task              == task
        new_assignee      == assignee
        previous_assignee == was(task.assignee)
        assigned_by       == actor
        assigned_at       == now

    failures:
      TaskNotAssignable: StateError
        condition: task.status == Done
                   or task.status == Archived

      ProjectArchived: StateError
        condition: task.project.status != Active


  operation StartTask
    inputs:
      task:  Task
      actor: User

    requires:
      task.status   == Open
      task.assignee == actor

    ensures:
      task.status     == InProgress
      task.updated_at == now

    side_effects:
      emit TaskStarted with:
        task       == task
        started_by == actor
        started_at == now

    failures:
      NotAssigned: AuthorizationError
        condition: task.assignee != actor

      AlreadyStarted: ConflictError
        condition: task.status == InProgress

      TaskNotOpen: StateError
        condition: task.status != Open


  operation CompleteTask
    inputs:
      task:  Task
      actor: User

    requires:
      task.status   == InProgress
      task.assignee == actor

    ensures:
      task.status     == Done
      task.updated_at == now

    side_effects:
      emit TaskCompleted with:
        task         == task
        completed_by == actor
        completed_at == now

    failures:
      NotAssigned: AuthorizationError
        condition: task.assignee != actor

      TaskNotInProgress: StateError
        condition: task.status != InProgress


  operation AddComment
    inputs:
      task:   Task
      author: User
      body:   Text

    requires:
      body is not empty
      task.status != Archived

    ensures:
      Comment created with:
        task    == task
        author  == author
        body    == body

    side_effects:
      emit CommentAdded with:
        task      == task
        comment   == created Comment
        author    == author
        added_at  == now

    returns:
      Comment

    failures:
      EmptyComment: ValidationError
        condition: body is empty

      TaskArchived: StateError
        condition: task.status == Archived

      CommentTooLong: ValidationError
        condition: body length > TaskPolicy.maximum_comment_length
        context:
          length:  Count
          maximum: Count


  --- queries ---

  @query GetProjectTasks
    find:  Task
    where: Task.project == project
    and:   Task.status != Archived
    order: Task.priority descending,
           Task.created_at descending

  @query GetMyTasks
    find:  Task
    where: Task.assignee == user
    and:   Task.status == Open
           or Task.status == InProgress
    order: Task.priority descending,
           Task.due_date ascending

  @query GetTaskComments
    find:  Comment
    where: Comment.task == task
    order: Comment.created_at ascending

  @query GetOverdueTasks
    find:  Task
    where: Task.due_date < now
    and:   Task.status == Open
           or Task.status == InProgress
    order: Task.due_date ascending

  @query GetTaskById
    find:    Task
    where:   Task.id == id
    returns: one


  --- events ---

  event TaskCreated
    task:       → Task      [required]
    project:    → Project   [required]
    assignee:   → User      [optional]
    created_by: → User      [required]
    created_at: Timestamp   [required]

  event TaskAssigned
    task:              → Task  [required]
    new_assignee:      → User  [required]
    previous_assignee: → User  [optional]
    assigned_by:       → User  [required]
    assigned_at:       Timestamp [required]

  event TaskStarted
    task:       → Task    [required]
    started_by: → User    [required]
    started_at: Timestamp [required]

  event TaskCompleted
    task:         → Task    [required]
    completed_by: → User    [required]
    completed_at: Timestamp [required]

  event CommentAdded
    task:     → Task    [required]
    comment:  → Comment [required]
    author:   → User    [required]
    added_at: Timestamp [required]


  --- extensions ---

  @extend index
    query: Task by project              [frequent]
    query: Task by assignee             [frequent]
    query: Task by project, status      [frequent]
    query: Task by assignee, status     [frequent]
    query: Task by due_date             [range]
    query: Comment by task              [frequent]

  @extend computed
    Project.active_task_count =
      count(Task
            where Task.project == Project.id
            and Task.status != Archived)

    Project.completed_task_count =
      count(Task
            where Task.project == Project.id
            and Task.status == Done)
```

## Part Six — Quick Reference

### Core Constructs

| Construct | Purpose |
|---|---|
| `entity` | Define a data structure |
| `enum` | Define a named value set |
| `type` | Define a custom domain type |
| `operation` | Define a behavioral action |
| `@query` | Define a data retrieval |
| `event` | Define something that happened |
| `on Event` | Subscribe to an event |
| `use X from Y` | Reference another domain |
| `@extend index` | Declare query performance hints |
| `@extend computed` | Define derived fields |
| `@extend audit` | Declare explicit audit tracking |

### Operation Sections

| Section | Required | Purpose |
|---|---|---|
| `inputs` | No | Parameters the operation receives |
| `requires` | No | Preconditions that must hold |
| `ensures` | Yes | Postconditions guaranteed on success |
| `side_effects` | No | Events emitted, notifications sent |
| `returns` | No | What is given back to the caller |
| `idempotent` | No | Safe to run multiple times |
| `failures` | No | What can go wrong and why |

### Special Values

| Value | Meaning |
|---|---|
| `was(field)` | Value of field before operation ran |
| `now` | Current timestamp |
| `null` | Absence of a value |

### Aggregate Functions

| Function | Returns | Field Type Required |
|---|---|---|
| `count(Entity where ...)` | Count | Any entity |

| `min(Entity.field where ...)` | Same as field | Comparable |
| `max(Entity.field where ...)` | Same as field | Comparable |
| `exists(Entity where ...)` | Boolean | Any entity |
| `all(Entity where ... such_that ...)` | Boolean | Any entity |
| `none(Entity where ... such_that ...)` | Boolean | Any entity |

## Part Seven — Design Principles

**The spec is the contract.** The client approves the specification before any code is written. The h doc they read and the s doc the compiler executes from are generated from the same source. What the client approved is what was built.

**Documentation cannot drift.** The h doc and s doc are generated together and maintained in sync by the AI. No human is responsible for keeping them aligned. The sync is structural.

**The compiler owns implementation decisions.** The spec declares what. The compiler decides how. Database choice, framework, query optimization strategy, retry implementation — these are compiler concerns, not spec concerns.

**History is the default.** Data is append-only unless explicitly marked transient. Memory is cheap. Lost history is not recoverable.

**Errors are citizens.** Failures are not afterthoughts. Every operation declares what can go wrong, under what conditions, with what context, and whether the caller can retry. The error hierarchy makes failure handling composable.

**Domains are independent.** Domains communicate through events, not direct calls. One domain failing does not cascade to others. Anti-fragility is architectural, not disciplinary.

**Extensions are the growth surface.** The core language is small and stable. New capabilities are added as named extensions. Existing spec files never break when new extensions are added.
